# Improve Agent Follow Through

## 背景

本次对话暴露出一个协作流程问题：用户已经明确表达“按文档来”“补充到规则里”“不要让我反复提醒”，但代理仍多次只在聊天中回答，没有主动同步到对应文档或常驻规则。

具体表现：

- 回答了 PowerShell 路径是否需要双引号，但没有立即更新刚创建的 `docs/codex-cli-usage-guide.md`。
- 补充了 `spec-workflow` skill，但没有主动判断同类常驻规则也应进入 `AGENTS.md`。
- 对低风险读取命令过度请求确认，违背了已确认规格中“低风险操作可直接执行”的约定。
- 用户追问后才补规则，缺少“回答后判断是否应沉淀”的自动动作。

## 目标

- 明确哪些信息应该自动沉淀到文档或规则。
- 明确哪些场景必须主动问用户，而不是沉默跳过或只聊天回答。
- 明确 `AGENTS.md`、`spec-workflow` skill、普通 `docs/` 文档之间的分层。
- 让代理在后续任务中更主动，但不越权执行高风险操作。

## 非目标

- 不修改 Go 代码。
- 不调整业务重构方案。
- 不改变“先文档后执行”的基础规则。
- 不把所有临时聊天内容都写进文档，避免文档噪声。

## 影响范围

预计需要更新：

- `AGENTS.md`
- `.agents/skills/spec-workflow/SKILL.md`
- `docs/codex-cli-usage-guide.md`

可能新增：

- `docs/agent-collaboration-guide.md`

是否新增独立协作指导文档，需要用户确认。

## 问题复盘

### 1. 没有识别“文档跟进”上下文

当用户问 PowerShell 路径写法时，问题明显属于刚创建的 Codex CLI 使用指导文档的补充内容。正确行为应是：

1. 先回答问题。
2. 判断该内容是否应同步到刚创建的文档。
3. 直接更新文档，或说明为什么不更新。

### 2. 没有主动做分层判断

当用户问“为什么不补到 AGENTS.md”时，说明该规则具有常驻价值。正确行为应是：

1. 说明分层原则。
2. 如果判断适合 `AGENTS.md`，直接补充。
3. 如果不适合，明确说明放在哪个 skill 或 docs 更合适。

### 3. 没有正确处理低风险操作

已确认的规格任务中，读取、搜索、格式化、测试、Wire 生成等低风险操作不应反复请求确认。正确行为应是：

1. 默认直接执行低风险操作。
2. 如果 sandbox 异常，先说明是工具环境失败，而不是操作风险。
3. 只有确实需要升级权限时才请求确认。

### 4. 没有主动追问用户偏好

当规则边界不清楚时，正确行为不是沉默或猜到底，而是提出少量明确问题让用户决策。

## 设计方案

### AGENTS.md 增加常驻规则

建议增加或细化以下规则：

- 当用户纠正代理行为、指出应沉淀文档或要求“以后不要再提醒”时，应判断是否需要更新 `AGENTS.md`、skill 或 docs；适合常驻的规则应进入 `AGENTS.md`。
- 如果用户表达“你要问我”“别默不作声”，遇到规则边界、执行风险或文档分层不清时，应主动提出 1-3 个具体问题。
- 已确认规格后的低风险操作应直接执行；不要把普通读文件、搜索、格式化、测试包装成需要确认的高风险动作。

### spec-workflow 增加执行判断规则

建议补充：

- 回答可复用命令、路径、平台差异、工作流规则后，应检查是否存在对应文档需要同步。
- 如果用户连续追问同一类缺口，应创建或更新流程改进规格，而不是只逐条被动修补。
- 当需要用户决策时，问题应聚焦在可执行选项上，不要问开放而空泛的问题。

### Codex CLI 文档补充操作习惯

建议补充：

- 推荐启动参数只是工具层配置，仓库规则仍由 `AGENTS.md` 和 skills 约束。
- 低风险操作不应反复确认。
- 如果 Codex 因 sandbox 失败请求确认，应区分“操作有风险”和“工具环境需要升级”。

2. 是否需要新增 `docs/agent-collaboration-guide.md`，专门记录“什么时候自动执行、什么时候主动问、什么时候沉淀文档”？已确认：需要新增，但关键规则仍必须在 `AGENTS.md` 和 skill 中引用，不能只依赖普通 docs。
