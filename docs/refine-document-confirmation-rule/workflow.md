# 细化执行阶段确认规则

## 要做什么

在现有“先文档后执行”规则基础上，新增一条执行阶段的确认边界：用户确认任务文档后，代理可以静默执行低风险操作，不需要对每个普通命令或文件修改反复询问；只有遇到危险操作、高影响操作或风险不确定的操作时，才需要暂停并请求用户确认。

## 为什么这么做

- 保留任务开始前的整体确认，避免代理误解需求后直接进入实现。
- 减少执行过程中的频繁确认，让常规开发动作可以连续完成。
- 把确认成本集中在真正可能造成损失的地方，例如数据破坏、外部系统变更、敏感信息风险和不可逆操作。
- 让代理在安全和效率之间有明确判断标准，而不是机械地每一步都询问。

## 如何做

1. 任务开始前仍然执行“先文档后执行”：创建 `docs/<task-name>/*.md`，写清楚目标、理由和任务列表，等待用户确认。
2. 用户确认后，进入执行阶段。
3. 执行阶段默认可以静默执行低风险操作。
4. 遇到危险操作、高影响操作或不确定风险时，暂停并向用户确认。
5. 如果执行过程中发现原计划不合适，仍然先回到对应文档记录原因和修正方案；只有修正影响范围较大或涉及危险操作时，才再次等待用户确认。

## 可静默执行的低风险操作

以下操作在任务文档已确认后，默认可以直接执行：

- 读取文件、搜索代码、查看 Git 状态和 diff。
- 修改当前仓库工作区内的普通源码、配置示例、测试和文档。
- 运行格式化、lint、单元测试、本地构建、代码生成等只影响本地工作区的命令。
- 新增、修改与任务相关的非敏感文档。
- 启动本地开发服务或本地测试命令，只要不会连接生产系统或修改外部数据。
- 对生成文件执行约定内的重新生成，例如 Wire、Proto 生成，但前提是任务文档已说明。

## 需要确认的危险或高影响操作

以下操作必须先询问用户：

- 改写数据库数据、执行迁移、清空表、导入覆盖数据，或连接非本地数据库执行写操作。
- 请求第三方 API、线上服务、生产环境或共享环境执行会修改数据的操作。
- 删除大量文件、递归删除、移动大目录，或执行难以恢复的文件系统操作。
- `git reset --hard`、强制推送、改写提交历史、丢弃未确认的工作区修改。
- 安装、升级、删除全局依赖，或修改系统级配置、用户目录配置、凭据文件。
- 发送消息、邮件、Webhook、支付、发布包、部署服务、重启线上服务。
- 输出、上传或写入未脱敏的 Token、密码、私钥、手机号等敏感信息。
- 网络下载或执行来源不明确的脚本、二进制文件。
- 任何风险判断不确定，或可能影响用户机器、外部系统、真实数据、生产环境的操作。

