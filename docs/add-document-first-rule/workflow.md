# 先文档后执行规则

## 要做什么

在 `AGENTS.md` 中新增一条常驻工作规则：除非用户明确要求跳过，否则代理在真正修改代码、配置、文档或执行有副作用的命令前，必须先在
`docs/` 下创建本次任务的说明文档，说明准备怎么做、为什么这么做、具体如何做，并等待用户确认后再开始执行。

建议规则名称：`先文档后执行`。

建议目录格式：

```text
docs/<task-name>/*.md
```

其中 `<task-name>` 使用能表达任务目标的英文短横线命名，例如：

```text
docs/add-document-first-rule/workflow.md
docs/refactor-service-setup/plan.md
docs/add-redis-setup/plan.md
```

## 为什么这么做

- 让每次改动先有可审阅的执行意图，减少代理误解需求后直接改代码的风险。
- 把“怎么做、为什么这么做、任务列表”沉淀到仓库文档里，后续可以追踪决策来源。
- 当执行过程中发现原计划不合适时，要求先回到文档记录偏差、原因和修正方案，再继续执行，避免无记录地改变方向。
- 对较复杂的重构、生成代码、配置接线、跨模块改动尤其有价值。

## 如何做

1. 收到任务后，先判断任务名称，并在 `docs/<task-name>/` 下创建或更新说明文档。
2. 文档至少包含：
    - `要做什么`：明确目标和范围。
    - `为什么这么做`：说明方案理由、约束和取舍。
    - `如何做`：列出可执行任务列表。
    - `确认状态`：标记是否已获得用户确认。
    - `执行记录`：执行中如有偏差，记录问题、原因、修正方案和时间。
3. 创建文档后，停止进入实际执行阶段，向用户说明文档路径并等待确认。
4. 用户确认后，按文档任务列表执行。
5. 如果执行过程中发现方案不对、约束不成立或需要改变路线，先回到文档补充：
    - 发现了什么问题。
    - 为什么原方案不合适。
    - 准备如何调整。
    - 为什么调整后的方案更合适。
6. 文档更新后，再继续执行；如果调整影响范围较大，应再次等待用户确认。
