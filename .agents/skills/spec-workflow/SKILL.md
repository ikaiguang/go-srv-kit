# Spec Workflow

## Overview

本 skill 用于在 Codex 中模拟并增强 Kiro specs 模式。

它只负责“规格先行”的协作流程，不负责仓库架构细节。修改 `go-srv-kit` 代码时，应同时遵循：

- `AGENTS.md`：最高优先级常驻规则
- `.agents/skills/go-srv-kit/`：仓库架构、Go 规范、Kratos、Wire、Proto、生成命令
- `.agents/skills/spec-workflow/`：规格文档、确认节点、执行记录和风险分级
- `docs/agent-collaboration-guide.md`：自动执行、主动询问、文档沉淀的具体判断示例

不要用本 skill 代替源码阅读，也不要用它覆盖 `AGENTS.md` 中的硬性规则。

## When To Use

当任务涉及以下行为时使用本 skill：

- 实现功能
- 修复 bug
- 重构代码
- 修改配置
- 修改 Wire 装配
- 修改 Proto 或生成代码流程
- 新增、修改或运行测试
- 执行格式化、构建、代码生成等有副作用命令
- 用户明确提到 specs、规格、计划、先文档、Kiro 模式

如果用户只是询问概念、解释代码或要求只读分析，通常不需要创建任务文档；但如果只读分析会推进为修改方案，应先回到本流程。

## Workflow

### 1. 识别任务名

根据用户目标生成简短的 kebab-case 任务名，例如：

- `add-spec-workflow-skill`
- `refactor-service-on-demand-modules`
- `fix-config-validation`

任务文档默认放在：

```text
docs/<task-name>/spec.md
```

### 2. 先创建规格文档

在执行任何修改、生成、格式化、测试或其他有副作用命令前，先创建或更新 `docs/<task-name>/spec.md`。

规格文档应说明：

- 要做什么
- 为什么这么做
- 不做什么
- 会影响哪些文件或模块
- 准备怎么做
- 怎么验收
- 有什么风险和回滚方式

### 3. 等待用户确认

创建或大幅更新任务规格后，停止实际执行，并明确告诉用户等待确认。

用户确认后，进入执行阶段。确认可以是“确认”“可以”“继续”“按这个做”等明确表达。

### 4. 按任务列表执行

执行阶段优先按 `spec.md` 中的任务列表推进。

低风险操作可以直接执行，包括：

- 读取和搜索文件
- 修改当前仓库内与任务相关的普通文件
- 运行本地格式化
- 运行本地测试
- 运行本地构建
- 运行仓库约定内代码生成

执行中保持最小改动，优先沿用现有实现和相邻代码风格。

### 5. 记录执行过程

执行过程中，应在 `spec.md` 的“执行记录”中记录关键节点：

- 何时开始执行
- 完成了哪些关键步骤
- 修改了哪些主要文件
- 执行过哪些验证命令
- 验证结果如何
- 是否偏离原方案

记录要简洁，但足够让后续恢复上下文。

### 6. 方案变化先回文档

如果发现原方案不适合，先更新 `spec.md`，写清楚：

- 发现的问题
- 原方案为什么不合适
- 准备怎么改
- 为什么这样改
- 是否扩大影响范围
- 是否引入新的风险

如果变化只是低风险细化，可以记录后继续。如果变化扩大影响范围、改变核心设计或引入中高风险，必须再次等待用户确认。

### 7. 完成后总结

任务完成后，最终回复应说明：

- 实际完成了什么
- 主要修改了哪些文件
- 运行了哪些验证
- 是否还有未完成事项或残余风险

同时确保 `spec.md` 的“执行记录”包含最终验证结果。

### 8. 处理用户纠正和连续追问

当用户纠正代理行为、指出“应该补文档/规则”、要求“不要让我反复提醒”或连续追问同类缺口时，不要只逐条被动回答。

应按以下顺序处理：

1. 判断该反馈是临时偏好、当前任务规则，还是跨任务常驻规则。
2. 常驻规则优先补到 `AGENTS.md`，细节补到本 skill 或对应 docs。
3. 当前任务规则补到 `docs/<task-name>/spec.md` 的执行记录或方案变更。
4. 可复用命令、路径、平台差异、协作习惯补到 `docs/*.md`。
5. 如果分层或风险不清楚，主动提出 1-3 个具体问题让用户决策。

如果用户已经明确要求“问我”，不要默不作声跳过不确定点；但低风险本地操作仍按已确认规格直接执行。

## Spec Template

默认使用单文件规格：

```md
# <Task Title>

## 背景

说明当前问题、用户目标、现有约束和为什么需要做这件事。

## 目标

- 列出本次要达成的结果

## 非目标

- 列出本次明确不处理的范围，避免任务膨胀

## 影响范围

- 列出预计新增、修改或需要重点阅读的文件和模块

## 方案

说明准备采用的实现方式、关键设计选择和与现有结构的关系。

## 任务列表

- [ ] 任务一
- [ ] 任务二
- [ ] 验证

## 验收标准

- 列出用户和代理都能判断是否完成的标准

## 风险与回滚

说明可能风险、规避方式和回滚办法。

## 执行记录

- 已创建任务规格文档，等待用户确认。
```

大任务可以拆分为多文件，但除非任务复杂度明显较高，否则优先使用单文件：

```text
docs/<task-name>/
├── 00-context.md
├── 01-requirements.md
├── 02-design.md
├── 03-tasks.md
└── 04-log.md
```

## Risk Levels

### Low Risk

用户确认规格后，低风险操作可以直接执行：

- 读取文件
- 搜索代码
- 修改当前仓库内与任务直接相关的普通源码或文档
- 运行 `gofmt`、`go test`、`go vet` 等本地验证命令
- 运行 `wire` 或 Makefile 中约定的本地生成命令
- 创建任务相关的 `docs/<task-name>/` 文档

### Medium Risk

中风险操作需要在规格文档中明确影响范围，必要时向用户解释：

- 批量重构
- 修改公共接口
- 修改配置结构
- 修改 Wire 装配
- 修改 Proto
- 调整跨模块依赖关系
- 修改测试基线或示例服务行为

如果中风险操作已经在用户确认的规格中写清楚，执行阶段可以继续推进。

### High Risk

高风险操作必须单独确认：

- 删除大量文件
- 改写 Git 历史
- 丢弃未确认修改
- 安装或修改系统级依赖
- 发布、部署或推送线上变更
- 请求第三方或线上 API 修改数据
- 处理未脱敏敏感信息
- 修改数据库数据或执行迁移到真实环境
- 任何风险不确定的操作

高风险确认要描述具体动作、影响范围和回滚方式。

## Coordination With Go Srv Kit Skill

当任务涉及 Go 代码、Kratos、DDD 分层、配置、Wire、Proto、生成命令、日志或错误处理时，同时使用 `.agents/skills/go-srv-kit/`。

推荐顺序：

1. 使用本 skill 创建或更新任务规格。
2. 用户确认后，按 `.agents/skills/go-srv-kit/` 读取对应技术参考。
3. 回到源码和相邻实现，做最小必要修改。
4. 更新 `spec.md` 执行记录。
5. 运行必要验证并总结结果。

## Practical Notes

- `AGENTS.md` 仍是最高优先级，若本 skill 与其冲突，以 `AGENTS.md` 为准。
- 如果用户明确要求跳过文档，可以不创建规格文档，但最终回复中应说明已按用户要求跳过。
- 如果用户只问“怎么做”“有没有办法”“能不能设计一下”，先回答方案即可，不要擅自修改文件。
- 如果用户的问题明显是在补充、纠正或完善刚创建的使用文档、规则文档或任务规格，应主动同步到对应文档；如果不能同步，先说明原因，不要只停留在聊天回答。
- 当回答涉及可复用的命令、路径、平台差异或仓库工作规则时，优先判断是否已有对应文档需要更新，避免让用户重复提醒。
- 已确认任务规格后，读文件、搜索、`gofmt`、`go test`、`wire`、`go list` 等低风险本地操作默认直接执行；只有越出当前 workspace、破坏性操作、线上或系统级影响、处理敏感信息、风险不确定时才询问确认。
- 如果用户确认后又提出新的方向，以最新用户消息为准，必要时更新规格文档后再继续。
- 不要为了形式而过度拆分小任务；规格的价值是降低歧义和留下上下文，而不是制造流程负担。
