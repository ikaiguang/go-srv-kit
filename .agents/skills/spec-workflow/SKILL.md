# Spec Workflow

## Overview

本 skill 只负责“先文档后执行”的协作流程，不负责仓库架构细节。修改代码时同时遵循：

- `AGENTS.md`：常驻硬规则
- `.agents/skills/my-project/`：仓库架构、Go 规范、生成命令和相邻实现入口
- 本 skill：规格文档、确认节点、执行记录和风险分级

任务文档是临时过程记录。长期规则应沉淀到 `AGENTS.md`、repo-local skill、模块 README 或稳定 docs。

## When To Use

当任务涉及以下行为时使用：

- 实现、修复、重构或生成代码
- 修改配置、Wire、Proto 或测试
- 运行格式化、构建、测试、生成器等有副作用命令
- 用户明确提到 specs、规格、计划、先文档、Kiro 模式

只读解释或方案讨论通常不用创建任务文档；一旦要推进到改文件或执行有副作用命令，应回到本流程。

## Workflow

1. 生成简短 kebab-case 任务名，默认文档路径为 `docs/<task-name>/spec.md`。
2. 先创建或更新规格文档，写清背景、目标、非目标、影响范围、方案、任务、验收标准、风险与回滚。
3. 创建或大幅更新规格后停止执行，等待用户确认。
4. 用户确认后按任务列表推进。低/中/高风险操作按下方 `Risk Levels` 处理。
5. 执行中在 `spec.md` 的“执行记录”记录关键步骤、修改文件、验证命令和结果。
6. 若方案变化扩大影响范围、改变核心设计或引入中高风险，先更新规格并再次等待确认。
7. 完成后总结实际修改、验证结果、未完成事项和残余风险；最终回复不要把临时任务文档作为长期入口。

## Feedback Handling

当用户纠正代理行为、要求沉淀文档或要求以后不要反复提醒时：

1. 判断反馈属于临时偏好、当前任务规则还是跨任务常驻规则。
2. 常驻规则补到 `AGENTS.md`；细节补到 repo-local skill、模块 README 或稳定 docs。
3. 当前任务规则补到任务文档执行记录或方案变更。
4. 如果分层或风险不清楚，主动提出 1-3 个具体问题。

## Spec Template

```md
# <Task Title>

## 背景

说明当前问题、用户目标、现有约束和为什么需要做。

## 目标

- 本次要达成的结果

## 非目标

- 本次明确不处理的范围

## 影响范围

- 预计新增、修改或需要重点阅读的文件和模块

## 方案

说明准备采用的实现方式、关键设计选择和与现有结构的关系。

## 任务列表

- [ ] 任务一
- [ ] 验证

## 验收标准

- 可判断任务完成的标准

## 风险与回滚

说明可能风险、规避方式和回滚办法。

## 执行记录

- 已创建任务规格文档，等待用户确认。
```

大任务可拆分多文件；普通任务优先使用单个 `spec.md`，避免流程负担。

## Risk Levels

### Low Risk

用户确认规格后可直接执行：

- 读取和搜索文件
- 修改当前仓库内与任务相关的普通源码或文档
- 运行 `gofmt`、`go test`、`go vet`、`go list`
- 运行仓库约定的本地生成命令，如 `wire` 或 Makefile 目标
- 创建或更新任务相关的 `docs/<task-name>/` 文档

### Medium Risk

需要在规格中明确影响范围；若已确认，可继续执行：

- 批量重构
- 修改公共接口或配置结构
- 修改 Wire 装配、Proto、测试基线或示例服务行为
- 调整跨模块依赖关系

### High Risk

必须单独确认：

- 删除大量文件
- 改写 Git 历史或丢弃未确认修改
- 安装或修改系统级依赖
- 发布、部署、推送线上变更
- 请求第三方或线上 API 修改数据
- 处理未脱敏敏感信息
- 修改数据库数据或执行真实环境迁移
- 任何风险不确定的操作

## Practical Notes

- 用户明确要求跳过文档时可以跳过，但最终回复中说明。
- 用户只问方案时先回答方案，不擅自改文件。
- 用户确认后又提出新方向，以最新消息为准，必要时更新规格后继续。
- 不为小任务过度拆分文档；规格的价值是降低歧义和留下上下文。
